EtherCAT 產品開發者的心得

發表於2023-09-01|更新於2024-08-30
|閱讀量:

# EtherCAT 介紹

EtherCAT 是一個開放架構，以 Ethernet 為基礎為現場總線應用提供支持，CAT 在 EtherCAT 中代表控制自動化技術 Control Automation Technology，EtherCAT 這個技術是由德國的 Beckhoff 公司所研發。

自動化應用通常對通信有一些特定的要求，如較短的資料更新時間、低通信抖動量以確保資料同步，同時成本也需要保持在合理範圍內。EtherCAT 系統的出現旨在取代傳統的總線接線系統，以更好地滿足這些需求。那麼，EtherCAT 是否成功地取代了傳統系統呢？

這個問題的答案可能會因應不同的使用情境而有所不同。實際的差異和效果可能會受到多種因素的影響，包括應用的性質、硬體和軟體的選擇，以及相關的維護需求。對我來說，我會根據我在產品開發方面的經驗以及客戶的反饋，來進行比較，以確定 EtherCAT 在特定情境下的優勢和不足。

因此，這篇文章並不是一篇嚴格的技術文章，而是我身為產品開發者和使用者的角度，來探討關於泛 EtherCAT 生態系統及其未來發展的看法。這樣的觀點可以提供有價值的見解，幫助我們更好地理解 EtherCAT 在自動化領域的地位和前景。

# EtherCAT 的系統價格

當談到 EtherCAT 系統的價格時可能會高於傳統的排線接線系統。這主要是因為 EtherCAT 從站的單價通常較高，每個從站都需要額外處理 EtherCAT 的通訊工作，這導致了額外的成本，例如額外通訊晶片或 MCU 升級需求，因此成本相應上升。

EtherCAT 控制器的價格則不一定比較昂貴。傳統的脈波式控制器在接線上有空間限制，最多只能控制 8 個軸，而 EtherCAT 控制器的單價雖然較高，但使用網路線進行接線，因此通常具有更高的軸數，可連接 32 至 64 個軸，這在多軸的應用情境下，EtherCAT 控制器在成本方面可能更具優勢，因此在多軸應用中，EtherCAT 控制器可以幫助節省成本，同時 EtherCAT 的接線相對簡便，這也有助於成本控制。

總結而言，當考慮 EtherCAT 的價格效益時，軸數是一個關鍵因素。如果你的機台需要控制的軸數超過 64 個，那麼使用 EtherCAT 有機會在降低成本，但對於小型機台，特別是八軸以內的應用，EtherCAT 會顯得昂貴，即使考慮到接線難度，也難以彌補成本差距。因此在大多數情況下，EtherCAT 相對於傳統接線來說可能會更昂貴。

# EtherCAT 的反應時間

EtherCAT 的反應時間是一個重要的議題，它涉及到系統的速度和效能。值得注意的是，EtherCAT 的速度主要指的是其反應速度，而不是資料傳輸量。EtherCAT 的資料傳輸量通常很小，所以如果你打算用它來傳送大型檔案，可能會感到失望。然而，即使將所有資源都集中在提高反應速度上，EtherCAT 的反應速度仍然無法與傳統的接線式控制系統相比。

有關 EtherCAT 反應時間被過分吹捧的問題，這可能是因為人們將其與 Ethernet 進行比較，實際情況是機台通常不使用 Ethernet，因此將其用作比較的對象不具有實際參考價值。EtherCAT 只是一種通訊協定和規範，它是多方約定的結果，並沒有引入實質性的技術突破。

有些客戶會詢問切換 EtherCAT 是否能使 UPH (每小時的產出) 上升，我可以負責任的說 UPH 只會下降不會提升，通常不會說的那麼直接，我會告知客戶 UPH 可能不會有什麼速度差異 (在理想的狀況下)。

有一個必須特別一提的問題，那就是機台的安全性，通常機台上會有不少顆，很大顆的紅色按鈕，有經驗的人就會知道那是 EMG (緊急停止)，當緊急狀況發生時馬上按下，立即停止機台運作減少人員的傷亡，如果將 EMG 實施在 Ethernet 上，可能存在安全性方面的疑慮，通訊中短暫的延遲，就會使勞工又少了幾根手指，但許多客戶可能仍會選擇簡單處理，因為設置雙迴路雙系統可能會失去簡化配線的優勢，而上報問題可能會引起管理層的不滿。

# EtherCAT 是縫合和怪

有時候我不禁懷疑 EtherCAT 的共識是否存在，或者它只是一種集體的幻覺。雖然有像 Beckhoff 這樣的開發和維護廠商，還有 EtherCAT 協會 ETG 等相關組織，看似一切都完備，但實際上這個領域充滿了爭議，令我疑惑真的有人認真在投入 EtherCAT，還是只是想賣掏金工具。

雖然 EtherCAT 制定了一套遊戲規則，但其內容卻經常出現不負責任的字句。文檔中不斷出現著「詳情請參考某某協定規範實作」或「問題請參考某某協定排除」等含糊指示。更令人困惑的是，這兩套規範看似毫無關聯，且經常存在著衝突。在市場上，產品可以隨便選擇其中一套實作，然後理所當然地宣稱它們符合標準。結果，同一個領域的產品充滿了千奇百怪的內容，但卻仍被認為符合標準。這種規範的不一致性，導致了市場上幾乎零兼容性的狀況，而導入一種新型產品就如同一次全新的客制化項目。

  • CAN Application Protocol over EtherCAT (CoE)
  • Servo Drive Profile according to IEC 61491-7-204 over EtherCAT (SoE)
  • Ethernet over EtherCAT (EoE)
  • File Access over EtherCAT (FoE)
  • ADS over EtherCAT (AoE)

客戶又開始奪命連環摳，真正搞事的產品不處理，整天想凹研發去客製化，特殊產品的文件寫的義正嚴詞，自我宣告為標準規範中的特殊用法，我試著按照文件的指示查找 EtherCAT 的相關資訊，但徒勞無功，花了好幾天的時間也沒有結果，這使我開始對人與人之間的信任產生懷疑，難道產品手冊都是亂寫的？但就在我快要放棄時，我突然想到了一種可能性。由於產品是日本製造所以還有日文版 EtherCAT 文件可以研究，日文版文件一開馬上就找到相關敘述，這讓我感到驚訝，日文版與英文版有著不少不同之處，總之內容不是逐字翻譯，我無法分辨誰才是正確資訊，不過肯定是個大坑。

最後，有一個有趣的發珼是，某些產品被歸為「其他」類別，儘管它們符合 EtherCAT 的標準規範，但它們未必實作了 EtherCAT 標準所需的功能。然而，這些產品卻完全符合「其他」類別的定義，因此它們仍被視為標準的「其他」類型

# EtherCAT 的應用難題

關於 EtherCAT 的應用難題，我們需要先來談 CoE，也就是 CANopen 中的 PDO 和 SDO，PDO 代表每個週期都要回傳資料 (通常週期時間是 1 毫秒)，而 SDO 則不需要每次都回傳，只有在需要的時候才執行。

EtherCAT 控制器可以連接多個軸，實際上這是指能夠連接 PDO 的資料長度。週期時間的降低導，致交握驗證等無效工作的時間倍增，使真正的有效資訊傳輸量占比極低。只要 PDO 長度稍微稱加，MCU 可能就無法負荷，可能導致連接失敗。高即時性不是無需代價的，它需要高昂的成本。

市面上產品宣傳的 64 軸和週期 0.25 毫秒的要求實際上是互斥的條件，無法同時實現。這是一個無解的難題，想要提高反應速度和連接站數，就必須提高 MCU 的規格，這將導致產品成本的上升。正如前面所提到的，EtherCAT 並沒有技術上的突破，它僅僅使用不同的資源分配方法。

# EtherCAT 不推薦投入

投入 EtherCAT 的決定不應該是草率的，這需要仔細考慮你的應用情境。如果你是大型機台的開發商，可以獲得優惠價格，而且你的應用需要資料傳輸，例如影像辨識，同時對速度要求不高，那麼 EtherCAT 可能是一個不錯的選擇。如果你的需求是中小型設備，尤其是在乎每小時產出 UPH 和成本高度敏感度，那麼選擇 EtherCAT 可能是一個錯誤。但無論如何，系統切換都是一項耗時的工作，絕非像宣傳中所說的那樣輕而易舉。根據我的經驗，大多數客戶在評估系統方面都花費了超過一年的時間。

如果你打算投入 EtherCAT 產品的開發，從站模組會是一個比較好的方向，最好與某個現有的主力控制器相容，並盡可能提高對其的兼容性。然而，如果你計劃自己開發控制器，那麼最好三思，除非你的公司對客製化有極高的熱情，並且能夠像微軟一樣每週更新，或者你打算大規模構建自己的生態系統。通訊協定眾多，EtherCAT 不是第一個，也不會是最後一個。每個人都希望主導制定規範獨吞大餅，因為保持優勢的最好方式就是迫使其他人按照你的規範行事。因此在決定投入 EtherCAT 之前，必須謹慎考慮你的需求和長遠目標。
