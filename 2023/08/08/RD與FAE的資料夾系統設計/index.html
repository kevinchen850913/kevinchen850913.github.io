RD與FAE的資料夾系統設計

發表於2023-08-08|更新於2023-10-29
|閱讀量:

# 我如何推行流程

我與部門主管討論的內容，動手前要一定先講清楚遊戲規則。

  • 資料只能存在於公司內網
  • 必須使用資料夾的形式
  • 我不會動前人的資料
  • 我不會干涉同事如何使用
  • 先測試，定期匯報，主管再決定是否上線

推廣流程其實十分容易
解決使用者的問題 => 使用者覺得好用一直用

失敗的原因也很簡單
造成使用者的困難 => 使用者能躲就躲

執行的方式就是，整理問題，提出改善方案，再整理回饋，再提出改善方案，周而復始，問題回饋狀況持續減少，此時流程就成熟了。

# 泛用型的資料夾

設計資料夾切勿使用加法思維，而是要聚焦於使用方式，可以列出常見使用情境，再反推回如何設計，要確保不要喪失原有功能，例如：禁止特定檔案可能是個好作法，杜絕規劃外的資料夾產生，要追加就要修改制度，提供一些多人資料夾系統泛用的項目。

  • 教育訓練區，部門總會有新人。
  • 工具軟體區，人總是會換電腦。
  • 內部測試區，工作總會需要合作。

# 設計 FAE 的資料夾

參考泛用型資料夾，聚焦 FAE 最重要的客戶與即時性，FAE 因工作性質總是四散在外，而最大的考驗就是如何快速且全面的交接資料，了解需求後資料夾也逐步成形。

  • 客戶專案區，客戶就是上帝，外出工作可掛勾地區做區分。
  • 產品資料區，有很多機密網路上找不到，務必要留備份。

我解決資料夾問題後，還偷偷的用 Git 管理客戶資料 (部門內沒人會)，離開客戶端回到公司後，連上內網資料夾，下個同步指令，休息喝杯咖啡，就又可以出門，完全不會有少帶資料的問題，下方圖片中就很明顯可以看到有人更新案件紀錄了。

# 設計 RD 的資料夾

參考泛用型資料夾，聚焦 RD 最重要的產品與專案，研發專案區要開案與結案時，會有固定的檔案需要清點，資料夾要能配合公司系統，更方便確認有無缺漏及版本控管，而參考文件多而繁雜，且分散在各個專案中難以控管，而且前人的使用經驗也難以傳承，後續用文件內聯的方式處理檔案重複問題，內聯至參考文件區，路徑最好用相對路徑，而且不要隨意修改。

  • 研發專案區，專案就是 KPI。
  • 產品資料區，文件真的不能不寫。
  • 參考文件區，別人的文件最好留底。

關於程式碼需要遠端庫備份，很簡單直接使用 git，由於公司沒有配套的措施，所以只簡單建立一個遠端庫，程式碼儲存區，主管交代只能用公司內網，所以大概很難再進行更動。

稍微整理資訊，資料夾系統就成形了，後續就只剩下一些較為旁枝末節的修改，我習慣會留一份 readme 記錄所有的修改細節，有文件才能避免糾紛。

# RD 資料夾系統實戰

依照上述的規劃，2021 年 6 月開始實際運作，我最初的設計是用心製圖來呈現與說明，後續才轉成資料夾開始實施，不過就找不到當時的心智圖，所以資料夾截圖來充數。

# RD 資料夾系統實戰 (兩年後)

經過兩年的實戰，至今仍足已應付絕大多數的工作需求，只追加了公司相關庶務區，該期間內其實沒有特別花時間維護，檔案都可以很直觀的找到正確的位置，且沒有出現個人資料夾與暫存區，可見該問題已成功地的得到改善。

公司相關庶務區：助理提出要追加放置公司行政流程文件，經過討論後追加的資料夾，而非憑空產生。

# 各區之說明

# 研發專案區

攸關 KPI 需較嚴謹的定義，參考公司規範文件設計之資料夾，反過來說就是系統不收的資料不能放，依照專案分類而非產品，若相依於產品而非專案，則不應該於此處更新，意思就是專案一結資料夾就不能再動了，後續刊誤或是各種修正只要沒另外開案，就是放在產品資料區，資料夾名稱就是專案號 + 專案名稱。

  • UA-2013-06_ABC
  • UD-2016-01_DEF

# 產品資料區

依照產品放置對外提供檔案，藉此把產品與專案去耦合，
例如：公版程式、軟體、韌體、手冊、安裝包…
補充：技轉專案結案後，檔案應在該區統一管理。

# 教育訓練區

放置教育訓練相關資料、SOP、手順文件等，使用連接至此方式避免文件多次出現。

# 參考文件區

放置對手產品文件、硬體零件資料等，使用連接至此方式避免文件多次出現，文件重複率太高 MIS 會投訴，這個資料夾我幾乎不會去碰，因為都是放電子零件資料，所以硬體工程師怎麼好作怎麼來，只要提供可以查到的型號或是超連結即可。

# 內部測試區

放置內部功能測試檔案，例如 : XX 實驗記錄、XX 數據表，
簡易保留開發記錄，提供未來回頭追蹤的可能性，或者可用此區作為內部傳檔案使用，總之就是至少要留底，因為檔案很多，所以要用年月日_功能。

  • 20230808_XX 實驗記錄
  • 20230908_重現 XX 錯誤

# 工具軟體區

放置開發使用工具，例如儀器光碟、工具電子檔案、小工具、金鑰等等，另外關於網路上可直接下載的檔案，須斟酌存放，雖然存放雷同檔案會浪費空間，但有時候就是會遇到下載連接失效或是版本匹配性問題，不要不信，下層資料夾就直接爆開，不過名稱還是要稍微管一下。

  • VisualStudio (保留舊版本，2017 後有下載器就不用特別留)
  • CCS、Keil-C 韌體環境 (差之毫釐，失之千里)
  • 各種小工具 (PDF 修改器，儀器應用程式)

# 程式碼儲存區

存放產品程式碼，使用 git 形式保存，所以當然也只能放 git 的版本庫，因為最少會用到所以擺在最後面，大概就一開始設定遠端時會用到，後來除非有人在遠端亂搞，不然不會需要開啟這個資料夾。

  • public，對外釋出的、正式的、有整理的。
  • 個人工號，自己的垃圾碼、基變前程式碼。
